# MiniNPUSimulator

Python으로 구현한 콘솔 기반 Mini NPU 시뮬레이터 프로젝트입니다.  
핵심 목표는 `Cross`와 `X` 패턴을 MAC(Multiply-Accumulate) 방식으로 비교하고, 사용자 입력 모드와 `data.json` 배치 분석 모드에서 동일한 판정 원리를 일관되게 적용하는 것입니다.

이 README는 단순 사용법 문서가 아니라, 아래 요구사항과 평가 기준을 실제 구현 관점에서 어떻게 충족하는지까지 설명하는 결과 리포트입니다.

## 1. 프로젝트 개요

### 1-1. 문제 정의

- 입력 패턴과 필터를 같은 좌표끼리 곱한 뒤 전부 더한 MAC 점수로 유사도를 계산합니다.
- 점수가 더 큰 필터를 “더 유사한 패턴”으로 판단합니다.
- 비교 대상은 `Cross`와 `X` 두 종류입니다.
- 동점 또는 부동소수점 수준의 미세한 차이는 `epsilon` 정책으로 처리합니다.

### 1-2. 제공 모드

| 모드 | 설명 | 핵심 요구사항 |
| --- | --- | --- |
| Mode 1 | 사용자가 3x3 필터 2개와 3x3 패턴을 직접 입력하거나 생성기로 패턴을 만듭니다. | 입력 검증, 재입력 유도, 점수/판정/시간 출력 |
| Mode 2 | `data.json`의 필터/패턴 케이스를 일괄 분석합니다. | 라벨 정규화, 크기 매칭, PASS/FAIL 집계, 케이스 단위 FAIL 처리 |

### 1-3. 제출물

- `main.py`: 메인 실행 파일
- `README.md`: 실행 방법 + 구현 분석 + 결과 리포트 + 시간 복잡도 분석
- `data.json`: 배치 테스트 입력 데이터
- `gen.py`: `data.json` 생성 스크립트

## 2. 실행 방법

### 2-1. 기본 실행

프로젝트 루트에서 아래 명령으로 실행합니다.

```bash
python main.py
```

실행 후 모드를 선택합니다.

```text
1. 사용자 입력 (3x3)
2. data.json 분석
```

### 2-2. Mode 1 실행 방식

1. `1`을 입력합니다.
2. `필터 A` 3줄, `필터 B` 3줄을 입력합니다.
3. 패턴 입력 방식을 선택합니다.
4. 직접 입력 또는 생성기(`Cross`/`X`)로 패턴을 준비합니다.
5. A/B 점수, 판정, 평균 MAC 시간, 성능 표를 확인합니다.

### 2-3. Mode 2 실행 방식

1. `2`를 입력합니다.
2. `main.py`와 같은 폴더의 `data.json`을 자동 로드합니다.
3. 각 케이스별 `PASS/FAIL`, 실패 사유, 성능 분석 표, 총합 요약을 출력합니다.

### 2-4. data.json 위치

- `data.json`은 `main.py`와 같은 디렉터리에 있어야 합니다.
- 파일이 없거나 JSON 문법이 깨져 있으면 프로그램은 안내 메시지를 출력하고 종료합니다.

### 2-5. 데이터 재생성

필요하면 아래 명령으로 `data.json`을 다시 생성할 수 있습니다.

```bash
python gen.py
```

### 2-6. 테스트 환경

| 항목 | 값 |
| --- | --- |
| OS | macOS 26.3.1 (a) |
| Python | 3.13.2 |
| 실행 위치 | 프로젝트 루트 디렉터리 |

파일별 역할은 다음과 같습니다.

## 4. 제약 사항 충족 분석

아래 표는 문제에서 요구한 핵심 제약 사항을 실제 구현에 대응시킨 것입니다.

| 요구사항 | 구현 방식 | 충족 근거 |
| --- | --- | --- |
| MAC 연산은 반복문으로 직접 구현 | `mac()`, `mac_flat()`에서 중첩 반복문으로 직접 곱셈/누적 수행 | `numpy`, 벡터화, 내장 행렬 연산을 사용하지 않음 |
| Mode 1 입력 최소 검증 | `parse_row_input()`이 열 개수, 숫자 파싱, 0/1 값 여부를 검사 | 형식 오류 시 `ValueError` 발생 후 `input_matrix()`에서 재입력 유도 |
| Mode 2에서 오류가 나도 전체 중단 금지 | `analyze_pattern_case()`가 케이스 단위 예외를 잡아 `FAIL` 결과 dict 반환 | 한 케이스 실패가 전체 배치 실행을 멈추지 않음 |
| 점수 비교는 epsilon 기반 | `EPSILON = 1e-9`, `decide_ab()`, `decide_label()`에서 사용 | 미세한 부동소수점 차이로 인한 오판정 방지 |
| README에 결과 리포트 포함 | 본 문서 6장, 7장, 8장에 결과/실패 원인/복잡도/병목 분석 명시 | 평가 항목 9~20 대응 |

### 4-1. MAC 연산은 왜 반복문으로 직접 구현했는가

이 과제의 핵심은 “NPU의 가장 단순한 곱셈-누산 흐름을 직접 구현하는 것”입니다.  
그래서 `mac(pattern, filt)`는 아래와 같은 순서를 그대로 따릅니다.

```text
total = 0
for i in range(N):
    for j in range(N):
        total += pattern[i][j] * filt[i][j]
return total
```

이 방식은 다음 장점이 있습니다.

- 각 좌표에서 어떤 곱셈이 일어나는지 추적이 쉽습니다.
- 시간 복잡도 `O(N²)`를 설명하기 쉽습니다.
- 예외 상황(행/열 크기 불일치)을 직접 통제할 수 있습니다.
- 벡터화 라이브러리를 쓰지 않으므로 과제 제약을 정확히 준수합니다.

### 4-2. Mode 1 입력 검증이 최소가 아니라 실질적인 이유

문제에서 요구한 최소 검증은 “행/열 개수, 숫자 파싱”이지만 현재 구현은 그보다 한 단계 더 엄격합니다.

- 한 줄의 원소 개수가 정확히 3개인지 검사
- 각 원소가 숫자로 변환 가능한지 검사
- 사용자 입력 모드에서는 값이 `0` 또는 `1`인지 추가 검사

즉, 잘못된 입력은 단순히 무시되지 않고, “어떤 이유로 틀렸는지” 안내한 뒤 같은 줄을 다시 입력하게 만듭니다.

### 4-3. Mode 2가 중단되지 않는 이유

`data.json`에는 의도적으로 잘못된 케이스가 포함되어 있습니다.  
이때 전체 프로그램이 죽으면 배치 평가에 치명적입니다.

현재 구현은 다음 순서로 방어합니다.

1. 최상위 JSON 구조를 먼저 검사합니다.
2. 필터는 미리 정규화하고, 사용할 수 없는 필터는 경고만 남깁니다.
3. 각 패턴 케이스는 `analyze_pattern_case()` 내부에서 독립적으로 분석합니다.
4. 예외가 나면 그 케이스만 `FAIL` 처리하고 다음 케이스로 넘어갑니다.

이 설계 때문에 크기 불일치, 숫자 아님, 알 수 없는 라벨 같은 오류가 있어도 전체 총합 집계는 끝까지 출력됩니다.

### 4-4. epsilon 정책이 필요한 이유

부동소수점은 10진수를 2진수로 완벽히 표현하지 못하는 경우가 많습니다.  
그래서 이론적으로 같아야 하는 두 값도 실제 계산 결과는 아주 미세하게 다를 수 있습니다.

현재 구현은 다음 정책을 사용합니다.

- `abs(score1 - score2) < 1e-9` 이면 동점으로 간주
- Mode 1에서는 `판정 불가`
- Mode 2에서는 `UNDECIDED`

이 정책이 없으면 “실제로는 같은 점수인데 한쪽이 조금 크게 보이는” 오판정이 발생할 수 있습니다.

## 5. 구현 요약

### 5-1. 전체 실행 흐름

### Mode 1

1. 모드 선택
2. 필터 A/B 입력
3. 입력 검증 실패 시 같은 줄 재입력
4. 패턴 직접 입력 또는 생성
5. `mac(pattern, filter_a)`, `mac(pattern, filter_b)` 계산
6. `decide_ab()`로 A/B/판정 불가 결정
7. `measure_mac_time()`으로 MAC 실행 시간 측정
8. 성능 표 출력

### Mode 2

1. `data.json` 로드
2. `filters`, `patterns` 최상위 구조 검증
3. 필터 라벨 정규화 및 크기별 그룹화
4. 패턴 키에서 크기 `N` 추출
5. 같은 크기의 필터 집합과 비교
6. `PASS/FAIL` 판정
7. 성능 표 및 총합 출력

### 5-2. 라벨 정규화 방식

입력 데이터는 서로 다른 표기 방식을 가질 수 있습니다.

| 원본 표기 | 내부 표준 라벨 |
| --- | --- |
| `+`, `cross`, `Cross` | `Cross` |
| `x`, `X` | `X` |

정규화를 별도 함수로 분리한 이유는 다음과 같습니다.

- 라벨 처리 규칙을 한 곳에서 관리할 수 있습니다.
- Mode 1, Mode 2, 생성기, 비교 로직이 같은 표준 라벨을 공유합니다.
- 새로운 라벨이 추가돼도 분기문을 여러 군데 고칠 필요가 없습니다.
- 잘못된 라벨을 빠르게 검출할 수 있습니다.

### 5-3. 패턴 크기와 필터 선택 방식

Mode 2에서는 패턴 키 이름에서 크기를 읽어 적절한 필터를 선택합니다.

예시:

- `size_5_1` -> 크기 `5`
- `size_13_2` -> 크기 `13`
- `size_25_1` -> 크기 `25`

이 정보를 이용해 내부 딕셔너리에서 다음처럼 매칭합니다.

```text
size_13_2
-> size = 13 추출
-> filters_by_size[13]["Cross"]
-> filters_by_size[13]["X"]
```

이 방식이 중요한 이유는 패턴과 필터 크기가 다르면 MAC 자체가 성립하지 않기 때문입니다.

### 5-4. MAC 점수가 높을수록 더 유사하다고 볼 수 있는 이유

현재 데이터는 0/1 기반 패턴입니다.

- `1 x 1 = 1` 이므로, 패턴과 필터가 같은 위치에서 동시에 1인 칸은 점수에 기여합니다.
- `1 x 0 = 0`, `0 x 1 = 0`, `0 x 0 = 0` 이므로, 겹치지 않는 칸은 점수에 기여하지 않습니다.

따라서 MAC 점수는 사실상 “두 도형이 얼마나 많이 같은 위치에서 겹치는가”를 세는 값입니다.  
겹치는 1의 개수가 많을수록 구조적으로 더 비슷하므로, 더 높은 점수를 얻은 필터를 정답 후보로 선택하는 것이 합리적입니다.

### 5-5. 동점 처리 정책

Mode 1과 Mode 2는 동점 처리의 표현이 다릅니다.

| 모드 | 동점 출력 | 이유 |
| --- | --- | --- |
| Mode 1 | `판정 불가` | 사용자가 보는 인터랙티브 결과라서 자연어 메시지가 더 직관적임 |
| Mode 2 | `UNDECIDED` | 배치 분석 결과에서 기계적으로 읽기 쉬운 표준 상태값이 필요함 |

Mode 2에서는 `UNDECIDED`를 곧바로 `FAIL`로 집계합니다.  
이유는 배치 테스트의 목적이 “정답 라벨과 예측 라벨이 일치하는지”를 보는 것이기 때문입니다.  
정답을 특정하지 못한 경우도 평가 관점에서는 실패입니다.

### 5-6. 시간 측정 경계

시간 측정은 I/O를 포함하지 않고, `MAC 함수 호출 구간`만 감쌉니다.

즉, 아래 요소는 제외됩니다.

- 사용자 입력 대기 시간
- 행렬 출력 시간
- JSON 파일 읽기 시간
- 요약 리포트 출력 시간

포함되는 것은 오직 아래 구간입니다.

```text
start = perf_counter()
score = mac(pattern, filter)
end = perf_counter()
```

이렇게 경계를 잡아야 연산 자체의 복잡도와 측정값을 직접 연결할 수 있습니다.

## 6. 재현성

이 절의 목적은 “실제로 실행했을 때 어떤 결과가 나와야 정상인지”를 명확히 기록하는 것입니다.

### 6-1. Mode 1 정상 재현 예시

입력:

```text
모드: 1

필터 A
0 1 0
1 1 1
0 1 0

필터 B
1 0 1
0 1 0
1 0 1

패턴 입력 방식: 2
생성 패턴: Cross
```

정상 기대 결과:

- `A 점수: 5.0`
- `B 점수: 1.0`
- `판정: A`
- 평균 MAC 시간과 성능 표 출력

실제 실행 예시:

```text
A 점수: 5.0
B 점수: 1.0
연산 시간(평균/10회, 단일 MAC 기준): 0.000731 ms
판정: A
```

### 6-2. Mode 1 동점 재현 예시

입력:

```text
모드: 1

필터 A
0 1 0
1 1 1
0 1 0

필터 B
1 0 1
0 1 0
1 0 1

패턴 직접 입력
1 1 1
1 1 1
1 1 1
```

정상 기대 결과:

- `A 점수: 5.0`
- `B 점수: 5.0`
- `판정: 판정 불가`
- `판정 불가 사유: |A-B| < 1e-09`

실제 실행 예시:

```text
A 점수: 5.0
B 점수: 5.0
판정: 판정 불가
판정 불가 사유: |A-B| < 1e-09
```

### 6-3. Mode 1 오류 입력 재현 예시

아래와 같은 잘못된 입력이 들어오면 프로그램은 중단되지 않고 다시 입력을 요구합니다.

잘못된 입력 예시:

- 열 개수 부족: `0 1`
- 숫자 아님: `1 0 a`
- 허용되지 않은 값: `2 0 1`

실제 안내 메시지 예시:

```text
입력 형식 오류: 각 줄에 3개의 숫자를 공백으로 구분해 입력하세요.
다시 입력해주세요.
```

또는

```text
입력 값 오류: 사용자 입력 모드에서는 0 또는 1만 입력할 수 있습니다.
다시 입력해주세요.
```

즉, 요구사항의 “오류 시 재입력을 유도한다”는 조건이 실제 동작으로 재현됩니다.

### 6-4. Mode 2 배치 분석 재현 예시

입력:

```text
모드: 2
```

현재 `data.json` 기준 정상 기대 결과:

- 총 테스트: `11개`
- 통과: `6개`
- 실패: `5개`
- 각 케이스별 PASS/FAIL과 실패 사유 출력

실제 요약 예시:

```text
총 테스트: 11개
통과: 6개
실패: 5개
```

현재 데이터셋의 실패 케이스는 의도적으로 포함되어 있으므로, `0 FAIL`이 아니라 `5 FAIL`이 정상입니다.

## 7. 결과 리포트

### 7-1. 전체 결과 요약

현재 `data.json`에는 총 11개의 패턴 케이스가 있습니다.

| 구분 | 개수 |
| --- | --- |
| 총 테스트 | 11 |
| PASS | 6 |
| FAIL | 5 |

PASS 6개는 모두 “정상 Cross/X 패턴이 해당 크기의 정답 필터와 일치하는 케이스”입니다.

- `size_5_1`
- `size_5_2`
- `size_13_1`
- `size_13_2`
- `size_25_1`
- `size_25_2`

### 7-2. FAIL 케이스 원인 분석

이 프로젝트에서 FAIL이 0이 아닌 이유는 구현이 잘못돼서가 아니라, `data.json`에 검증용 음수 테스트가 의도적으로 들어 있기 때문입니다.

| 케이스 | 관찰 결과 | 실패 원인 | 프로그램이 중단되지 않는 이유 |
| --- | --- | --- | --- |
| `size_5_3` | `predicted=Cross`, `expected=X` | 입력 패턴은 Cross인데 정답 라벨을 일부러 X로 넣은 정답 불일치 케이스 | 단순 비교 불일치이므로 FAIL만 기록 |
| `size_5_4` | `UNDECIDED` | 모든 값이 1인 5x5 패턴이라 Cross와 X 점수가 모두 9로 동일 | 동점은 `UNDECIDED` 후 FAIL 처리 |
| `size_5_5` | 크기 검증 실패 | 키는 `size_5_*`인데 실제 패턴은 3x3 | 케이스 단위 검증 실패로 처리 |
| `size_5_6` | 예외 발생: unknown label | `expected`가 `triangle`이라 정규화 불가 | 예외를 잡아 FAIL 사유로 변환 |
| `size_5_7` | 숫자 아님 검증 실패 | 패턴 내부에 문자열 `"a"` 포함 | 숫자 타입 검사 후 FAIL 처리 |

### 7-3. 왜 “0 FAIL”이 오히려 이상한가

현재 데이터셋은 정상 동작 여부만 보는 것이 아니라, 오류 처리까지 검증하도록 설계되어 있습니다.  
따라서 아래 다섯 상황을 모두 FAIL로 잡아내는 것이 오히려 정답입니다.

- 정답 라벨 불일치
- 허용오차 내 동점
- 크기 불일치
- 알 수 없는 라벨
- 숫자가 아닌 값 포함

즉, 현재 결과 `11개 중 6 PASS / 5 FAIL`은 구현이 요구사항을 정확히 수행한 결과입니다.

## 8. 성능 분석 및 시간 복잡도

### 8-1. 샘플 측정 결과

아래 표는 현재 환경에서 `Mode 2`를 실행했을 때 얻은 한 번의 샘플 측정값입니다.  
절대 시간은 환경에 따라 조금 달라질 수 있지만, 증가 추세는 동일해야 합니다.

| 크기 | 평균 시간(ms) | 연산 횟수(N²) |
| --- | ---: | ---: |
| 3x3 | 0.000742 | 9 |
| 5x5 | 0.001354 | 25 |
| 13x13 | 0.006254 | 169 |
| 25x25 | 0.020529 | 625 |

### 8-2. 왜 O(N²)인가

패턴 크기가 `N x N`이면 방문해야 하는 원소 수는 총 `N²`개입니다.  
현재 MAC 구현은 각 원소마다 정확히 한 번의 곱셈과 한 번의 누적 덧셈을 수행합니다.

따라서 MAC 1회의 연산량은 다음처럼 볼 수 있습니다.

```text
곱셈 N²회 + 덧셈 N²회 + 인덱스 접근/반복 제어
= 상수배를 무시하면 O(N²)
```

분류 1회는 Cross 필터와 X 필터를 각각 한 번씩 비교하므로, 전체 판정 비용은 대략 `2 x O(N²)`이며, 빅오 표기에서는 여전히 `O(N²)`입니다.

### 8-3. 측정 결과와 O(N²) 근거 연결

연산 횟수와 시간 증가를 함께 보면 다음 경향이 보입니다.

| 비교 | 연산 횟수 증가 | 시간 증가 | 해석 |
| --- | ---: | ---: | --- |
| 3x3 -> 5x5 | 9 -> 25, 약 2.78배 | 0.000742 -> 0.001354, 약 1.82배 | 작은 입력에서는 타이머 오버헤드 영향이 큼 |
| 5x5 -> 13x13 | 25 -> 169, 약 6.76배 | 0.001354 -> 0.006254, 약 4.62배 | 입력이 커질수록 `N²` 증가 경향이 더 분명해짐 |
| 13x13 -> 25x25 | 169 -> 625, 약 3.70배 | 0.006254 -> 0.020529, 약 3.28배 | 측정값이 연산량 증가와 거의 같은 방향으로 커짐 |

즉, 절대 배수가 완전히 일치하지는 않지만, 입력 크기 증가에 따라 시간도 대체로 `N²`에 비례해 커지는 경향이 확인됩니다.  
이 차이는 파이썬 인터프리터 오버헤드, 함수 호출 비용, 캐시 영향, 매우 작은 시간 단위 측정 잡음 때문입니다.

### 8-4. 보너스: 2차원 vs 1차원 비교

현재 구현은 보너스 분석으로 2차원 리스트 기반 MAC와 1차원 평탄화 MAC도 비교합니다.

샘플 결과:

| 크기 | 2D 평균(ms) | 1D 평균(ms) | 속도비(2D/1D) |
| --- | ---: | ---: | ---: |
| 3x3 | 0.000742 | 0.000458 | 1.618x |
| 5x5 | 0.001354 | 0.000696 | 1.947x |
| 13x13 | 0.006254 | 0.003850 | 1.624x |
| 25x25 | 0.020529 | 0.016717 | 1.228x |

1차원 방식이 종종 더 빠른 이유는 다음과 같습니다.

- 리스트 중첩 인덱싱이 줄어듭니다.
- 메모리 접근이 더 순차적입니다.
- 파이썬 레벨에서 간접 참조 횟수가 감소합니다.

다만 과제의 핵심 요구사항은 “반복문 기반 MAC 구현”이므로, 1차원 비교는 분석용 보너스일 뿐 기본 구현을 대체하지는 않습니다.

## 9. 심층 설계 분석

### 9-1. MAC 로직의 전체 흐름

MAC 처리의 구조는 아래와 같습니다.

1. 패턴과 필터의 행 수가 같은지 확인
2. 각 행의 열 수가 같은지 확인
3. 좌표 `(i, j)`를 순회
4. `pattern[i][j] * filt[i][j]`를 누적
5. 총합 반환

이 구조는 하드웨어 NPU의 기본 개념인 “곱하고 누적한다”를 가장 직접적으로 보여줍니다.

### 9-2. 라벨 정규화를 함수로 분리한 이유와 장점

정규화를 여러 군데 흩어놓으면 다음 문제가 생깁니다.

- 어떤 곳은 `+`를 지원하고 어떤 곳은 지원하지 않을 수 있음
- 예외 메시지가 일관되지 않을 수 있음
- 새로운 라벨 추가 시 수정 위치를 놓치기 쉬움

그래서 별도 함수 하나를 “단일 진실 공급원(single source of truth)”으로 두는 것이 유지보수에 유리합니다.

### 9-3. 부동소수점 오차와 epsilon의 필요성

부동소수점은 내부적으로 근삿값을 사용합니다.  
특히 값의 누적이 많아지면 아주 작은 오차가 생길 수 있고, 직접 `==` 비교를 하면 동점이어야 할 케이스를 다르게 판정할 위험이 있습니다.

`epsilon` 정책은 “사람이 보기엔 같은 값인 두 점수”를 기계도 같은 값으로 취급하게 만드는 안전장치입니다.

### 9-4. 새로운 라벨이 추가될 때의 확장 순서

예를 들어 `Triangle` 라벨을 추가한다고 가정하면, 확장 순서는 다음이 적절합니다.

1. `normalize_label()`에 새 표기와 표준 라벨 매핑 추가
2. 필요하면 패턴 생성기 함수 추가
3. Mode 2 필터 구조에 `Triangle` 필터 추가
4. 판정 로직을 다중 클래스 비교 형태로 확장
5. 출력 형식과 요약 리포트에 새 라벨 반영
6. 정상 케이스와 실패 케이스를 테스트 데이터에 추가

이 순서를 따르면 “입력은 되는데 판정이 안 되는 상태”를 줄일 수 있습니다.

### 9-5. 대규모 패턴 처리 시 시간/메모리 문제

패턴 크기가 매우 커지면 두 가지 문제가 생깁니다.

| 구분 | 문제 | 이유 |
| --- | --- | --- |
| 시간 | MAC가 느려짐 | 각 케이스마다 `N²` 순회를 해야 함 |
| 메모리 | 큰 2차원 리스트 저장 비용 증가 | 필터와 패턴 모두 `N²` 공간 필요 |

개선 아이디어는 다음과 같습니다.

- 1차원 평탄화로 메모리 접근 단순화
- 희소(sparse) 패턴이면 1의 좌표만 저장
- 큰 데이터셋은 파일 전체를 한 번에 올리지 않고 스트리밍 처리
- 케이스 단위 병렬 처리
- 같은 필터를 재사용할 때 전처리 결과 캐싱

### 9-6. 직렬 처리와 병렬 처리 관점의 병목

현재 병목은 거의 전부 `MAC 반복문`입니다.

- 직렬 처리 병목: 각 케이스에서 두 필터에 대해 `N²` 순회를 반복하는 부분
- 병렬화 후보: 케이스별 분석, 또는 Cross/X 두 필터 비교를 분리하는 부분

다만 현재 구현은 순수 Python CPU 바운드 코드이기 때문에 `threading`의 이득은 제한적일 수 있습니다.  
큰 데이터셋에서 병렬화를 하려면 `multiprocessing` 또는 네이티브 확장이 더 적합합니다.

- 반복문 기반 MAC 구현
- Mode 1 입력 검증 및 재입력 유도
- Mode 2 케이스 단위 FAIL 처리
